## Шаг 1. Заполнить `.env`

Откройте файл `.env` (если его нет — скопируйте `.env.example`). Где взять каждое значение:

| Переменная | Где взять | Нужно для |
|------------|-----------|-----------|
| `BOT_TOKEN` | Телеграм → **@BotFather** → `/mybots` → ваш бот → *API Token* | бот (уже есть) |
| `OWNER_TG_ID` | Телеграм → **@userinfobot** → пришлёт ваш `Id` | бот (уже есть) |
| `ACCOUNTANT_TG_ID` | тот же @userinfobot, от лица бухгалтера | бот (уже есть) |
| `DATABASE_URL` | Supabase → Project Settings → Database → Connection string (URI) | бот (уже есть) |
| `ANTHROPIC_API_KEY` | console.anthropic.com → API Keys | AI (уже есть) |
| **`WEBAPP_URL`** | `https://<ваш-домен>` — адрес, где будет открываться Mini App (см. Шаг 4) | Mini App |
| `ROBOKASSA_MERCHANT_LOGIN` | Robokassa → Мои магазины → идентификатор магазина | приём Robokassa |
| `ROBOKASSA_PASSWORD` | Robokassa → Технические настройки → **Пароль #2** | подпись Robokassa |
| `PRODAMUS_SECRET_KEY` | Prodamus → Настройки → **секретный ключ** | подпись Prodamus |
| `TOCHKA_CLIENT_ID` | Точка → Интеграции и API → зарегистрировать OAuth-приложение | синхронизация Точки |
| `TOCHKA_CLIENT_SECRET` | там же | синхронизация Точки |

> Платёжные источники — заполняйте **только те, что используете**. Можно начать с одного.

Проверка: `npm run preflight` → раздел «Ядро» должен быть весь ✓.

## Шаг 3. Зарегистрировать Mini App в @BotFather

В Телеграме напишите **@BotFather**:

1. `/mybots` → выберите вашего бота.
2. **Bot Settings** → **Menu Button** → **Configure Menu Button**.
3. Пришлите URL вашего Mini App: `https://<ваш-домен>` (тот же, что `WEBAPP_URL`).
4. Пришлите текст кнопки, например: `Аналитика`.

Готово — теперь в чате с ботом слева снизу появится кнопка, открывающая Mini App.
Команда `/app` в боте тоже откроет приложение (это уже в коде).

> ⚠️ URL обязан быть **https://** с валидным сертификатом — иначе Telegram не откроет. Сертификат ставится автоматически на Шаге 4.

## Шаг 4. Развернуть на сервере (Beget VPS)

Подробная инструкция — в [deploy/README.md](deploy/README.md). Коротко, первый раз:

```bash
ssh <user>@<IP-сервера>
sudo mkdir -p /opt/finassist && sudo chown $USER:$USER /opt/finassist
cd /opt/finassist && git clone <repo_url> .

sudo bash deploy/setup-server.sh          # Node 20, nginx, certbot, PM2

# домен в конфиге nginx
sudo nano /etc/nginx/sites-available/finassist   # заменить app.example.com → ваш домен
sudo nginx -t && sudo systemctl reload nginx

sudo nano /etc/finassist/.env             # вставить секреты из вашего .env

# DNS A-запись домена должна указывать на IP сервера, затем:
sudo certbot --nginx -d <ваш-домен>       # выпустить HTTPS-сертификат

bash deploy/deploy.sh                      # сборка + миграции + запуск
curl -s https://<ваш-домен>/api/health     # {"status":"ok",...}
```

Обновление в будущем — просто `bash deploy/deploy.sh`.

## Шаг 5. Прописать webhook-адреса в кабинетах платёжек

После того как домен заработал по HTTPS (Шаг 4), укажите адреса уведомлений:

| Источник | Куда вставить | Адрес |
|----------|---------------|-------|
| **Robokassa** | Личный кабинет → Технические настройки → **Result URL** (метод: POST) | `https://<ваш-домен>/api/webhooks/robokassa` |
| **Prodamus** | Настройки → URL для уведомлений (webhook) | `https://<ваш-домен>/api/webhooks/prodamus` |
| **Точка** | ничего вставлять не нужно — синхронизация сама опрашивает банк каждые 30 мин | — |

> Robokassa и Prodamus **присылают** платежи на эти адреса (push). Точка — наоборот, бот сам её опрашивает (pull, по OAuth).

Проверка Robokassa: после тестовой оплаты в `pm2 logs finassist` появится `robokassa_webhook_signature_ok`.

## Шаг 6. Финальная проверка

- [ ] `npm run preflight` — «Ядро» всё ✓.
- [ ] `curl https://<домен>/api/health` → `{"status":"ok"}`.
- [ ] В боте кнопка/команда `/app` открывает Mini App, виден дашборд.
- [ ] (если подключили) тестовая оплата Robokassa/Prodamus → транзакция появилась в БД, в логах `signature_ok`.
- [ ] `pm2 logs finassist` — нет повторяющихся ошибок.

## Важно про подписи платёжек (нужна боевая проверка)

Алгоритмы проверки подписи Robokassa (MD5) и Prodamus (HMAC-SHA256) реализованы по официальной документации и покрыты юнит-проверкой, но **точность сериализации Prodamus и имена полей выписки Точки стоит сверить на первом реальном платеже** — детали в `feature-spec-integrations-sync.md` (раздел ASSUMPTION). Если первый webhook отклонён с `bad signature` — пришлите содержимое лога (без ключей), поправим за 5 минут.
