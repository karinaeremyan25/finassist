# FEATURE_SPEC.md — Справочник структуры спецификации фичи

> Reference-документ: что Claude генерирует для FEATURE_SPEC.md.
> Используйте для валидации результата и формулирования требований — не для ручного заполнения.

---

## Философия этого документа

**Вы не пишете FEATURE_SPEC.md вручную.** Вы описываете Claude AI, какую фичу хотите добавить — а Claude генерирует структурированную FEATURE_SPEC.md с миграциями, API, UI и edge cases.

Зачем тогда этот справочник?

1. **Валидация** — понять, все ли разделы Claude заполнил (миграция, API, UI, edge cases, scope файлов)
2. **Корректировка** — если нет RLS, нет примеров, не перечислены файлы — знать как попросить доработать
3. **Коммуникация** — говорить с Claude на языке "добавь edge case для лимитов" вместо "это неполная"
4. **Контроль scope** — убедиться, что Claude не трогает ничего лишнего в существующем проекте

**Правильный workflow:**
```
Вы → описываете фичу голосом/текстом (что надо, для кого, зачем)
Claude AI → генерирует FEATURE_SPEC.md по этому формату
Вы → используете этот справочник чтобы валидировать результат
Claude → дорабатывает разделы по вашим замечаниям
Claude Code → читает FEATURE_SPEC.md и реализует фичу в существующем проекте
```

---

## Когда использовать FEATURE_SPEC вместо полной SPEC.md

Когда применим этот формат:
- Добавляете новый модуль в существующий проект
- Клиент попросил фичу, которой нет в исходной спецификации
- Расширяете MVP после первого релиза

Когда НЕ применим (используйте полную SPEC.md):
- Строите проект с нуля
- Изменения затрагивают больше 50% кодовой базы — это уже рефакторинг

**Правило:** Одна FEATURE_SPEC = одна фича. Не смешивайте несколько фич в одном документе — попросите Claude сгенерировать отдельный документ на каждую.

## Что Claude вернёт для TaskFlow: AI-чат

Ниже — пример того, что Claude сгенерирует, если вы опишете ему фичу "AI-чат для управления задачами" в проекте TaskFlow. Используйте как образец для сравнения с тем, что получите в своём проекте.

```markdown
# Feature: AI-чат для управления задачами

> **Проект:** TaskFlow
> **Дата:** 2026-04-10
> **Приоритет:** High
> **Оценка:** 1 день

---

## 1. User Story

**Как** фрилансер с 20+ задачами,
**я хочу** управлять задачами через чат на естественном языке,
**чтобы** не искать нужную задачу на доске, а просто спросить или дать команду.

### Сценарий использования
1. Пользователь нажимает иконку чата (💬) в правом нижнем углу
2. Открывается чат-панель (Sheet справа, ширина 400px)
3. Пользователь пишет: "Покажи все просроченные задачи"
4. AI отвечает списком задач с дедлайном в прошлом, сгруппированных по проекту
5. Пользователь пишет: "Перенеси задачу 'Отправить макет' на завтра"
6. AI обновляет дедлайн задачи и подтверждает: "Готово! Дедлайн задачи 'Отправить макет логотипа' обновлён на 11 апреля."
7. Изменение мгновенно отражается на канбан-доске (Realtime)

### Критерии приёмки
- [ ] Чат открывается за < 500ms
- [ ] AI понимает русский язык
- [ ] AI может: показать задачи (фильтр), создать задачу, обновить статус, обновить дедлайн
- [ ] AI НЕ может: удалить задачу, удалить проект (только через UI)
- [ ] Ответ AI приходит за < 3 секунды
- [ ] История чата сохраняется в рамках сессии (очищается при закрытии)

## 2. Изменения в базе данных

### Новые таблицы

-- История сообщений AI-чата
CREATE TABLE chat_messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('user', 'assistant')),
  content TEXT NOT NULL,
  tool_calls JSONB, -- сохраняем вызовы инструментов для аудита
  tokens_used INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_chat_messages_user_id ON chat_messages(user_id);
CREATE INDEX idx_chat_messages_created_at ON chat_messages(created_at);

ALTER TABLE chat_messages ENABLE ROW LEVEL SECURITY;

CREATE POLICY "chat_messages_select_own"
  ON chat_messages FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "chat_messages_insert_own"
  ON chat_messages FOR INSERT
  WITH CHECK (auth.uid() = user_id);
```

## 3. Изменения в API

### Новые эндпоинты

#### `POST /api/chat`
**Описание:** Отправка сообщения AI-ассистенту с function calling для управления задачами
**Авторизация:** Требуется (Bearer token)

**Тело запроса:**
```json
{
  "message": "Покажи все просроченные задачи",
  "history": [
    {
      "role": "user",
      "content": "Привет"
    },
    {
      "role": "assistant",
      "content": "Привет! Я могу помочь управлять задачами. Спросите о задачах или дайте команду."
    }
  ]
}
```

**Ответ 200:**
```json
{
  "data": {
    "message": "У вас 3 просроченные задачи:\n\n**Проект: Редизайн сайта**\n- Отправить макет логотипа (дедлайн: 8 апреля)\n- Согласовать цветовую палитру (дедлайн: 5 апреля)\n\n**Проект: Лендинг для клиники**\n- Подготовить мобильную версию (дедлайн: 9 апреля)",
    "tool_calls": [
      {
        "name": "get_tasks",
        "arguments": { "filter": "overdue" },
        "result": { "count": 3, "tasks": ["..."] }
      }
    ],
    "tokens_used": 847
  }
}
```

**Ответ 429 (лимит):**
```json
{
  "error": {
    "code": "RATE_LIMIT",
    "message": "Лимит сообщений: 50 в день. Попробуйте завтра или подключите подписку."
  }
}
```

### AI Tools (function calling)

Ассистент использует Claude API с tools:

| Tool | Описание | Параметры |
|------|----------|-----------|
| `get_tasks` | Получить задачи с фильтрацией | filter: "all" / "overdue" / "today" / "week"; project_id?: UUID; status?: string |
| `create_task` | Создать задачу | title: string; project_id: UUID; deadline?: string; priority?: string |
| `update_task` | Обновить задачу | task_id: UUID; title?: string; status?: string; deadline?: string; priority?: string |
| `get_projects` | Получить список проектов | status?: "active" / "archived" |

### Изменения в существующих эндпоинтах
Изменения в существующих эндпоинтах не требуются.

## 5. Business Logic

### Правила
- Бесплатный план: 50 сообщений в день. При превышении — показать toast с предложением подписки
- AI НЕ может удалять данные (задачи, проекты) — только создавать и обновлять
- AI имеет доступ ТОЛЬКО к данным текущего пользователя (user_id передаётся через контекст, не через параметры)
- Максимальная длина сообщения пользователя: 1000 символов
- История чата передаётся с каждым запросом (последние 20 сообщений), не хранится на сервере между сессиями

### Интеграции
- **Сервис:** Anthropic Messages API
- **Модель:** Claude Sonnet (быстрые ответы, низкая стоимость)
- **System prompt:** "Ты — TaskFlow AI, помощник для управления задачами. Отвечай кратко, на русском. Используй инструменты для работы с задачами. Не выдумывай задачи — используй только реальные данные пользователя."
- **Max tokens:** 500 (ответ ассистента)
- **При ошибке API:** Retry 1 раз через 2 секунды. Если повторная ошибка — показать сообщение пользователю

## 6. Edge Cases

| # | Ситуация | Поведение системы |
|---|----------|-------------------|
| 1 | Пользователь просит удалить задачу через чат | AI отвечает: "Удаление задач доступно только через интерфейс. Перейдите на доску проекта." |
| 2 | Пользователь спрашивает про чужие задачи | AI не видит чужих данных (RLS). Отвечает: "Задач, соответствующих запросу, не найдено." |
| 3 | AI не понял команду пользователя | AI отвечает: "Не совсем понял. Вот что я умею: показать задачи, создать задачу, обновить статус или дедлайн." |
| 4 | Anthropic API недоступен | Показать в чате: "AI-ассистент временно недоступен. Попробуйте через несколько минут." |
| 5 | Пользователь отправляет пустое сообщение | Кнопка отправки заблокирована, если input пустой (disabled) |
| 6 | Пользователь отправляет 1001+ символов | Счётчик символов под полем ввода, красный при превышении, отправка заблокирована |
| 7 | 50 сообщений исчерпаны на бесплатном плане | Поле ввода заблокировано. Текст: "Вы исчерпали лимит на сегодня. Подключите подписку для безлимитного чата." |

## Чек-лист валидации (после генерации)

Когда Claude сгенерировал FEATURE_SPEC.md — пройдитесь по этому чек-листу. Если что-то не так, попросите Claude доработать конкретный раздел.

```
[ ] 1. User Story описывает конкретный сценарий (не абстракцию)
[ ] 2. Критерии приёмки измеримы (время, количество, точные условия)
[ ] 3. SQL-миграция готова к выполнению (или явно написано "не требуется")
[ ] 4. Новые таблицы имеют RLS-политики
[ ] 5. API-эндпоинты с примерами запросов И ответов (включая ошибки)
[ ] 6. UI-компоненты с тремя состояниями (Loading, Empty, Error)
[ ] 7. Edge cases — минимум 5 сценариев
[ ] 8. Список затронутых файлов указан явно (scope изменений)
[ ] 9. Указаны новые файлы И изменяемые файлы раздельно
[ ] 10. НЕТ TODO, placeholder, "определить позже"
[ ] 11. Интеграции описаны с обработкой ошибок (retry, fallback)
[ ] 12. Бизнес-правила описаны с реакцией системы на их нарушение
```

**Если найдена проблема — промпт для исправления:**
```
В FEATURE_SPEC.md раздел [X] неполный: [что не хватает — например: 
"новая таблица без RLS" / "не перечислены затронутые файлы" / "edge cases меньше 5"].
Доработай этот раздел по справочнику 2-12-FEATURE_SPEC_TEMPLATE.md.

## Workflow: как создать FEATURE_SPEC.md за 15 минут

```
ШАГ 1: Сформулируйте запрос на фичу в свободной форме
  ↓ (что надо, для кого, какую боль решает, как примерно должна работать)
  
ШАГ 2: Откройте Claude AI (web) или Claude Code
  ↓ Загрузите 3 файла:
  • Ваше описание фичи
  • 2-12-FEATURE_SPEC_TEMPLATE.md (этот справочник)
  • CLAUDE.md существующего проекта (чтобы Claude понимал контекст)
  
ШАГ 3: Промпт:
  "Сгенерируй FEATURE_SPEC.md для этой фичи по формату из справочника.
   Учти архитектуру существующего проекта из CLAUDE.md.
   Заполни все 7 разделов: user story, миграция с RLS, API с примерами,
   UI с 3 состояниями, бизнес-логика, минимум 5 edge cases, список файлов."
  
ШАГ 4: Получите результат
  ↓ Claude вернёт FEATURE_SPEC.md на 100-300 строк
  
ШАГ 5: Валидация (этот документ)
  ↓ Пройдитесь по чек-листу выше
  
ШАГ 6: Если нужны правки — попросите Claude
  "В разделе [X] не хватает [Y]. Доработай."
  
ШАГ 7: Передайте FEATURE_SPEC.md в Claude Code
  ↓ Промпт: "Прочитай FEATURE_SPEC.md и реализуй фичу. 
     Не трогай существующий функционал кроме указанных файлов."
     
ШАГ 8: Claude Code применяет фичу
  ↓ Миграция → API → компоненты → git commit
```

**Итог:** вы не написали FEATURE_SPEC.md руками. Вы описали идею фичи, валидировали сгенерированный документ, передали его в Claude Code. Это и есть роль архитектора при итерации продукта.
