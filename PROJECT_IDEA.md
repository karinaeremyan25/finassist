# FinAssist — AI-ассистент по финансам в Telegram

> Telegram-бот на Node.js + Anthropic API (Claude), который ведёт учёт
> доходов и расходов по двум юрлицам (ООО «Ассургина» и ИП Карина Еремян)
> и двум направлениям бизнеса (Курс ДПО «Психология здоровья» и Клуб
> «Метанойя»), показывает чистую прибыль по направлениям, ведёт фонды
> накоплений и отвечает на финансовые вопросы голосом цифр.
> Используется собственником, бухгалтером и руководителем проекта.

---

## ⚠ Допущения, требующие подтверждения

Эти пункты приняты по умолчанию, чтобы зафиксировать MVP.
Карина проверяет и при необходимости поправляет:

1. **Руководитель проекта — один человек на оба направления**
   (Курс ДПО + Метанойя). Если их двое и каждый видит только своё
   направление — раздел «Аудитория» переписывается, в SPEC.md
   добавляется фильтрация по направлению.
2. **Руководитель проекта НЕ видит личные траты Карины с карты ИП**
   и НЕ видит данные по второму направлению (если их два).
   Видит только агрегированные отчёты по своему направлению:
   выручка, расходы по бизнес-категориям, чистая прибыль.
3. **Руководитель проекта может вносить расходы по своему направлению
   через бота** (например, оплатил подрядчика — записал в боте).
   Бухгалтер потом верифицирует.
4. **Точный формат выгрузок Продамуса** (CSV / Excel / PDF) уточняется
   у бухгалтерии и фиксируется в SPEC.md перед началом сборки.

## 1. Проблема

### Текущая ситуация (AS-IS)
Поступления приходят с Продамуса каждые 1–2 рабочих дня и попадают
на расчётные счета ООО «Ассургина» и ИП Карина Еремян. Часть оплат
идёт наличными и на личные карты (включая зарубежную карту в иностранной
валюте). Бухгалтер выгружает выписки из личного кабинета Продамуса,
видит траты по карте ИП (включая личные расходы Карины), получает
от Карины информацию о зарубежной карте и ведёт учёт расходов
и доходов в Excel или собственной программе. Карина получает
финансовую картину от бухгалтера эпизодически — обычно к моменту
уплаты налогов или при возникновении ощущения «что-то не так
с деньгами».

### Боль
- Финансовая картина живёт у бухгалтера, а не у Карины: чтобы
  понять «сколько чистой прибыли по Метанойе за апрель», нужно
  дождаться отчёта или запросить выгрузку — это растягивается на дни
- Нет разделения по двум направлениям бизнеса в реальном времени:
  бухгалтер ведёт общий учёт, разрез «Курс ДПО vs. Метанойя»
  собирается отдельно и нерегулярно
- Нет дисциплины фондов: налоги, резерв, развитие, личные накопления
  смешаны в одном потоке денег. Решения о тратах принимаются
  по ощущению, а к дате налогов формируется кассовое напряжение
- Нет ранних сигналов: проседание направления, рост категории
  расходов, риск кассового разрыва замечаются постфактум —
  когда уже больно
- Бухгалтер тратит 4–8 часов в месяц на классификацию и сведение
  тех же транзакций, которые Карина и руководитель проекта могли бы
  заносить в момент совершения с автоматической классификацией
- Стратегические решения (масштабировать направление, нанять,
  поднять цену, закрыть программу) принимаются без актуальных цифр

### Цена проблемы
Один пропущенный сигнал об убыточности направления — это 2–4 месяца
работы команды вхолостую: сотни тысяч рублей зарплат и упущенной
выгоды, которые могли пойти на работающее направление. Отсутствие
фонда налогов создаёт риск кассового разрыва в квартал уплаты —
изъятие денег из оборота или экстренный заём. Запаздывание
с финансовой картиной на 2–4 недели регулярно приводит к решениям
по ощущению, которые потом приходится откатывать. Время бухгалтера
на ручную классификацию — 4–8 часов в месяц, оплачивается
из бюджета бизнеса.

## 2. Целевая аудитория

### Основная (ядро) — 3 роли с первого дня MVP

**Роль 1: Собственник (Карина Еремян)**
- **Кто:** владелец ИП и ООО «Ассургина», ведёт два образовательных
  направления по Германской новой медицине — Курс ДПО «Психология
  здоровья» и Клуб «Метанойя»
- **Что делает в боте:** записывает расходы и поступления в свободной
  форме (текст / голос), запрашивает аналитику по направлениям,
  принимает решения по распределению фондов
- **Что видит:** ВСЁ — включая свои личные траты и зарубежную карту
- **Где обитает:** Telegram (уже основной канал коммуникации)

**Роль 2: Бухгалтер**
- **Кто:** действующий бухгалтер Карины, ведёт учёт ИП и ООО
- **Что делает в боте:** выгружает выписки из Продамуса и пересылает
  файл боту, вносит банковские операции по обоим юрлицам, фиксирует
  данные по зарубежной карте (со слов Карины), верифицирует
  и при необходимости корректирует записи Карины и руководителя
- **Что видит:** ВСЁ — поступления, расходы по обоим юрлицам,
  личные траты Карины с карты ИП, операции по зарубежной карте,
  баланс фондов, отчёты по обоим направлениям

**Роль 3: Руководитель проекта**
- **Кто:** руководитель образовательных направлений (по умолчанию —
  один на оба направления; см. блок «Допущения»)
- **Что делает в боте:** записывает расходы по своему направлению
  (оплата подрядчиков, оборудование, реклама, маркетинг),
  запрашивает отчёты по направлению
- **Что видит:** агрегированные отчёты по своему направлению —
  выручка, расходы по бизнес-категориям, чистая прибыль, динамика.
  НЕ видит: личные траты Карины, зарубежную карту, цифры
  по другим направлениям

- **Размер группы:** 3 пользователя на старте, до 5 при появлении
  второго руководителя проекта
- **Платёжеспособность:** инфраструктурный бюджет до 3 000 руб./мес.
  (Anthropic API + VPS + БД), монетизация продукта не предусмотрена

### Антиаудитория (НЕ для них)
- Корпоративные финансовые отделы — им нужны 1С, Adesk, Финансист
  с многопользовательским доступом, сложными ролями и правами
- Малый бизнес с 1000+ транзакций в месяц — нужна полноценная
  бухгалтерская автоматизация, а не разговорный ассистент
- Пользователи без Telegram — продукт привязан к одному каналу
- Эксперты, ведущие учёт самостоятельно без бухгалтера — продукт
  спроектирован под трио «собственник + бухгалтер + руководитель»

## 3. Решение

### Формула ценности
Карина пишет боту о расходе в свободной форме, бухгалтер пересылает
выписку Продамуса, руководитель направления заносит свои траты →
бот разносит каждую транзакцию по юрлицу, направлению и категории,
ведёт фонды, отвечает на вопросы и шлёт алерты → у каждого участника
команды актуальная финансовая картина в Telegram, чистая прибыль
по двум направлениям видна в любой момент, к дате налогов фонд
сформирован, решения принимаются на цифрах.

### Ключевые функции (7 для MVP)

| # | Функция | Кто делает | Пользователь делает | Система делает | Результат |
|---|---------|-----------|---------------------|----------------|-----------|
| 1 | Загрузка выписки Продамуса | Бухгалтер | Пересылает боту файл выписки | Парсит файл, привязывает каждое поступление к юрлицу (ООО / ИП) и направлению (Курс ДПО / Метанойя), дедуплицирует | Все поступления в учёте за 1 минуту, дубли исключены |
| 2 | Запись операции в свободной форме | Карина, руководитель | Пишет/говорит: «Купила оборудование 45 000, ИП, Метанойя» | Распознаёт текст или голос, классифицирует по юрлицу + направлению + категории, уточняет недостающее одним вопросом | Транзакция записана за 15–30 секунд |
| 3 | Внесение операций по картам и наличным | Бухгалтер, Карина | Записывает банковские операции и операции по зарубежной карте (с указанием валюты) | Сохраняет в исходной валюте + пересчитывает в рубли по курсу ЦБ на дату операции | Полный учёт включая зарубежную карту и наличные |
| 4 | Аналитика по направлениям | Все три роли (по своему контуру доступа) | Запрашивает: «Прибыль по Метанойе за апрель» / «Сравни направления за квартал» | Считает выручку, расходы, чистую прибыль, маржинальность по направлению с учётом роли запрашивающего | Видно, какое направление зарабатывает, какое проседает |
| 5 | Фонды и распределение | Карина | После крупного поступления получает предложение: «Распределить 200 000 ₽? Налоги 25%, резерв 10%, развитие 15%, личное 50%» | Ведёт виртуальные фонды, показывает баланс каждого, предупреждает при просадке резерва | К дате налогов фонд сформирован, нет кассового разрыва |
| 6 | Динамика и алерты | Все три роли (по своему контуру) | Ничего — система сама | Раз в неделю: сводка по выручке/расходам, рост категорий > 30%, прогноз на конец месяца, алерт о низком фонде налогов перед уплатой | Проблема замечается за 2–4 недели до того, как станет острой |
| 7 | Финансовые рекомендации | Карина | Спрашивает: «Стоит ли запускать новый поток Курса ДПО?» / «Какой фонд недонабираю?» | Анализирует историю по направлению (выручка, расходы, маржа, сезонность) и баланс фондов, даёт рекомендацию с цифрами и обоснованием | Решение по бизнесу за 2 минуты диалога вместо 2 дней |

### Что НЕ входит в MVP
- Автоматическая интеграция с банками по API (открытое банковское
  API ООО — отдельный проект)
- Интеграция с API Продамуса для автоматической загрузки выписок —
  на старте только ручная пересылка файла бухгалтером.
  Подключается, если Продамус предоставит подходящий API
- Формирование официальной бухгалтерской отчётности и подача
  в ФНС — это работа человека-бухгалтера, бот её не заменяет
- Веб-интерфейс с дашбордами — на старте только Telegram
- Расчёт зарплат и удержаний по команде
- Прогнозирование на год вперёд и финансовое моделирование
- Многопользовательские чаты внутри бота — каждый пользователь
  работает с ботом в личной переписке

## 4. Ограничения

### Технические
- **Бюджет на инфраструктуру:** до 3 000 руб./мес.
  (Anthropic API + VPS под Node.js + база данных)
- **Стек:** Node.js + Telegram Bot API + Anthropic Messages API
  (Claude). Реализация — через Claude Code в режиме автономной
  сборки по SPEC.md
- **Интеграции на старте:** Telegram Bot API, Anthropic API,
  ручная пересылка выписок Продамуса бухгалтером.
  Точный формат файла Продамуса (CSV / Excel / PDF) уточняется
  у бухгалтерии перед началом SPEC.md
- **Хранение данных:** база на VPS под контролем Карины. Все
  финансовые данные — только на её серверах, без облаков третьих лиц
- **Требования к структуре данных:** все суммы хранятся в копейках
  (INTEGER); валюта зарубежной карты — в исходной валюте + пересчёт
  в рубли по курсу ЦБ на дату операции; категории расходов
  структурированы по бухгалтерской логике (прямые по направлению /
  общие операционные / налоги / личные); у каждой транзакции —
  ответственный (кто внёс) и статус верификации бухгалтером

### Временные
- **Дедлайн MVP:** 4–5 недель до момента, когда работают функции
  1–5 с тремя ролями. Функции 6–7 (алерты, рекомендации) —
  ещё 1–2 недели
- **Доступность разработчика:** Claude Code в режиме автономной
  сборки по SPEC.md, ревью и валидация — Карина в роли архитектора

### Бизнесовые
- **Юрисдикция:** Россия. Учёт двух юрлиц — ООО «Ассургина»
  и ИП Карина Еремян, плюс личные средства собственника
- **Платежи в продукте:** не применимо (личный инструмент)
- **Доступ к боту:** только по telegram_id из whitelist (Карина,
  бухгалтер, руководитель проекта). Любые другие — отказ
- **Персональные данные:** в боте обрабатываются финансовые данные
  Карины и обезличенные суммы платежей клиентов из выписок Продамуса
  (без ФИО, если не нужны для учёта)

## 5. Критерии успеха

### MVP считается успешным, если в течение первого месяца после запуска:
1. **Полнота выручки:** 100% поступлений с Продамуса за месяц
   загружены в бота — нет «потерянных» платежей
2. **Скорость учёта расходов:** не менее 80% расходов за месяц
   записаны в боте в день их совершения (а не задним числом)
3. **Бухгалтер ведёт учёт В боте, а не параллельно в Excel** —
   главный показатель того, что MVP заменил собой текущий процесс,
   а не добавил третий учёт
4. **Точность чистой прибыли по направлениям:** в конце месяца
   отчёт «чистая прибыль по Курсу ДПО / по Метанойе / итого»
   совпадает с независимой ручной проверкой ±2%
5. **Фонд налогов:** к дате квартальной уплаты налогов фонд
   сформирован полностью, изымать из оборота не нужно
6. **Влияние на решения:** не менее 1 бизнес-решения в месяц принято
   на основе аналитики бота (фиксируется в свободной форме в боте)

### Метрики для отслеживания:
- **Активация:** в первую неделю — загружена первая выписка Продамуса
  бухгалтером + не менее 5 ручных расходов от Карины + не менее
  1 расхода от руководителя
- **Регулярность:** Карина — не менее 4 дней в неделю с активным
  взаимодействием. Бухгалтер — не менее 2 раз в неделю (в дни
  получения выписок). Руководитель — не реже 1 раза в неделю
- **Полнота учёта:** ≥ 95% поступлений и ≥ 80% расходов за месяц
  отражены в боте
- **Доверие к цифрам:** разница между отчётом бота и ручной сверкой
  бухгалтера — не более 2% по итогу месяца

## 6. Монетизация

Не применимо. Продукт — личный инструмент Карины и её команды
(собственник, бухгалтер, руководитель проекта). Коммерческой
монетизации не предусмотрено.
