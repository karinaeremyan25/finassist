# Telegram Mini App Architecture

## Цель
Добавить лёгкую веб-версию аналитической платформы FinAssist, которую пользователи открывают из Telegram как Mini App.

### Ограничения
- Не дублировать бизнес-логику между ботом и Mini App.
- Не использовать сложный менеджмент во фронте.
- Не интегрировать Anthropic во фронте.
- Все AI-вызовы выполняются только через бэкенд.
- Фронт получает уже готовые тексты от бэка.

## Что уже есть
- Персональный AI-агент в Telegram (Financial Director)
- Работает в проде на Beget VPS
- Загрузка выписок
- Автокатегоризация транзакций
- Smart Agent на Opus 4.8 (`claude-opus-4-8`)
- Conversation Memory + Prompt Caching
- Knowledge Graph
- Голос через Deepgram

## Что добавляется
- Платформа аналитики
- Визуализация данных
- Многопользовательский доступ

## Архитектурная концепция

### Общая идея
Mini App — это легкий пользовательский интерфейс, который работает как веб-приложение внутри Telegram, сохраняя всю логику и AI на backend. Фронтенд выступает слоем отображения и управления фильтрами, но не принимает решения и не выполняет AI-запросы.

### Принципы
- Backend-driven frontend.
- AI-процессинг остается на сервере.
- Минимальный frontend state: текущий пользователь, выбранные фильтры и локальный UI state.
- Авторизация через Telegram login / bot session, без дополнительных внешних сервисов.
- Данные выводятся в виде готовых текстов, KPI-карточек и chart-ready payload.
- Не хранить конфиденциальные данные в браузере.

## Компоненты

### Фронтенд
- Лёгкий SPA или традиционный набор страниц на React/Vite или Vanilla TS.
- Набор экранов:
  - Dashboard аналитики
  - Транзакции
  - Пользователи / доступ
  - Настройки фильтров
- UI-состояния:
  - Loading
  - Empty
  - Error
  - Ready

### Бэкенд
- Существующий Node.js + TypeScript сервер.
- PostgreSQL через postgres.js.
- Сервис для аналитики: агрегаты + готовые текстовые инсайты.
- Сервис для webapp session: проверка Telegram initData или токена.
- API endpoints для данных и готовых выводов.

### БД
- Повторное использование текущих таблиц транзакций, пользователей, фондов и направлений.
- При необходимости добавить таблицу логов Web App сессий / аудит.
- Сохранять все суммы в копейках, даты в UTC, soft delete через `deleted_at`.

## Data flow

1. Пользователь открывает Mini App из Telegram.
2. Telegram передает `initData` на клиент.
3. Клиент отправляет `POST /api/webapp/session` или `GET /api/users/me`.
4. Сервер верифицирует `telegram_id` и возвращает профиль пользователя.
5. Клиент запрашивает аналитические данные и инсайты.
6. Сервер собирает данные из БД, формирует готовые тексты, графики и возвращает JSON.
7. Клиент отображает dashboard.

## Источники данных и синхронизация (платформенный слой)

> Это инфраструктура приложения: данные попадают в БД независимо от Mini App и AI-наставника. Все потребители (дашборд, инсайты, AI-агент) только читают уже синхронизированные данные.

### Источники
- **Robokassa**: автоматическая синхронизация через REST API.
- **Prodamus**: автоматическая синхронизация через REST API.
- **Tochka (Точка Банк)**: синхронизация через OAuth 2.0 API (счета, выписки, балансы, операции).

### Расписание (cron, UTC)
- `*/30 * * * *` — одновременная синхронизация Robokassa, Prodamus и Tochka.
- `0 6 * * *` и `0 15 * * *` — генерация ежедневной финансовой сводки `generateDailyFinancialReport()` (утро и вечер).

### Правила
- Дедупликация по `external_id` с CHECK `external_id IS NOT NULL AND deleted_at IS NULL`.
- Все суммы в копейках, даты в UTC, soft delete через `deleted_at`.
- Rate limiting: максимум 3 одновременных синхронизации, очередь обработки.
- При ошибке синхронизации одного источника: логировать, отправить алерт, не блокировать работу остальных источников и Mini App.
- При невалидных credentials источника: отключить синхронизацию этого источника до исправления.

## Безопасность и защита данных

- **Credentials в environment variables**: `ROBOKASSA_MERCHANT_LOGIN`, `ROBOKASSA_PASSWORD`, `PRODAMUS_API_KEY`, `TOCHKA_CLIENT_ID`, `TOCHKA_CLIENT_SECRET`.
- Переменные окружения загружаются только на сервере, никогда не отправляются на фронтенд.
- API-ключи и пароли не логируются — только метаинформация (количество транзакций, timestamp, source). Strict-check в logger.
- Все запросы к Robokassa/Prodamus/Tochka идут через backend, фронт не имеет прямого доступа.
- Данные кешируются на backend (память или Redis), не передаются клиенту raw.
- SSL/TLS обязателен для всех внешних HTTP-запросов и внутри Mini App.
- Audit log: все синхронизации логируются с timestamp, source и количеством операций (без персональных данных).

## API

### Новый API

#### `POST /api/webapp/session`
- Описание: инициализация Mini App, проверка Telegram-сессии.
- Авторизация: требуется `initData` от Telegram.
- Запрос:
```json
{ "initData": "<telegram_init_data>" }
```
- Ответ 200:
```json
{
  "user": { "telegram_id": 12345, "name": "Карина", "role": "owner" },
  "entities": [ { "id": "...", "name": "ИП Карина Еремян" } ],
  "defaultPeriod": { "from": "2026-06-01", "to": "2026-06-10" },
  "features": ["analytics", "transactions", "users"]
}
```
- Ошибка 401: `{ "error": { "code": "unauthorized", "message": "Сессия не распознана" } }`

#### `GET /api/analytics/summary`
- Описание: сводка по выбранному периоду.
- Авторизация: `telegram_id`.
- Запрос: `?entity_id=&from=&to=&direction_id=`.
- Ответ 200:
```json
{
  "totalIncome": 12000000,
  "totalExpense": 4500000,
  "balance": 7500000,
  "fundStatus": {
    "taxFund": 900000,
    "reserveFund": 300000,
    "gratitudeFund": 142000,
    "creditFund": 284000,
    "profitFund": 1200000
  },
  "categoryBreakdown": [
    { "category": "Маркетинг", "amount": 1500000 },
    { "category": "Зарплата", "amount": 2000000 }
  ]
}
```

> **`fundStatus` — суммы в копейках.** `taxFund` и `reserveFund` берутся из балансов фондов (`funds`/`fund_transactions`). `gratitudeFund` и `creditFund` — **кастомные фонды проекта**, считаются из транзакций за период: `creditFund` = расходы на кредит мужа (`getLoanExpenseMetrics`: категория `exp_loan_husband` + описание ~«кредит»/«loan»), `gratitudeFund` = операции «благодарность» в Точке (`getGratitudeFundMetrics`: описание ~«благодар»/«благотвор»). Нужны для donut на Dashboard (см. `mini-app-design.md` §6.4): «Прибыль» = `totalIncome − totalExpense − taxFund − gratitudeFund − creditFund` (вычисляемый остаток, в БД не хранится).

#### `GET /api/analytics/charts`
- Описание: данные для графиков.
- Запрос: `?entity_id=&from=&to=&group_by=day|week|month`.
- Ответ 200:
```json
{
  "labels": ["01.06", "02.06"],
  "series": [
    { "name": "Доход", "data": [500000, 600000] },
    { "name": "Расход", "data": [200000, 250000] }
  ],
  "type": "line"
}
```

#### `GET /api/analytics/insights`
- Описание: готовые текстовые выводы.
- Запрос: `?entity_id=&from=&to=`.
- Ответ 200:
```json
{
  "insights": [
    { "title": "Рост выручки", "text": "Выручка за неделю выросла на 12% по сравнению с предыдущей." },
    { "title": "Фонд налога", "text": "Налоговый фонд сформирован на 74%, рекомендуем отложить дополнительные 15000 ₽." }
  ]
}
```

#### `GET /api/analytics/transactions`
- Описание: список транзакций с категорией и статусом.
- Запрос: `?entity_id=&from=&to=&limit=&offset=`.
- Ответ 200:
```json
{
  "transactions": [
    { "id": "...", "date": "2026-06-08T12:00:00Z", "description": "Оплата курса", "amount": 250000, "direction": "Курс ДПО", "category": "Доход" }
  ],
  "total": 42
}
```

#### `GET /api/webapp/users`
- Описание: список активных пользователей.
- Авторизация: требуется.
- Ответ 200:
```json
{
  "users": [
    { "telegram_id": 12345, "name": "Карина", "role": "owner", "last_seen": "2026-06-10T09:00:00Z" }
  ]
}
```

### Изменения в существующих API
- `GET /api/users/me`: может расшириться метаданными Mini App.
- `POST /api/transactions/import`: может добавлять `source: "webapp"`, но без изменения логики.

## UI

### Новые экраны
| Компонент | Путь/Расположение | Что показывает |
|-----------|-------------------|----------------|
| Dashboard | `/dashboard` | KPI, графики, карточки инсайтов |
| Transactions | `/transactions` | Таблица транзакций с фильтрами |
| Users | `/users` | Список пользователей и доступ |
| Settings | `/settings` | Выбор периода, юрлица и направления |

### Состояния
- Loading: спиннер или skeleton.
- Empty: "Нет данных за выбранный период" с подсказкой.
- Error: "Не удалось загрузить аналитику. Попробуйте позже.".

### UX-правила
- Без сложного state management.
- Фильтры отправляют запросы на сервер, приложение рендерит ответ.
- Вся текстовая аналитика приходит готовой.
- Карточки KPI должны быть кликабельны для переключения периодов.

## Business Logic

### Основные правила
- Вся бизнес-логика выполняется на backend.
- Frontend не делает AI-запросы и не хранит промежуточные промпты.
- Практически весь UI работает на данных из backend.
- Многопользовательский доступ определяется через `telegram_id` и таблицу `app_users`.
- Данные из Mini App должны соответствовать тем же правилам, что и бот: суммы в копейках, soft delete, даты UTC.

### Интеграции
- `services/analytics.ts`: агрегирует транзакции, вычисляет KPI, готовит тексты.
- `services/webapp.ts`: проверяет Telegram session и разрешает доступ.
- `services/claude.ts` и `services/classifier.ts`: остаются только на backend.

## Edge Cases

| # | Ситуация | Поведение системы |
|---|----------|------------------|
| 1 | Telegram initData не валиден | Ответ 401, сообщение "Сессия не распознана" |
| 2 | Нет транзакций за период | Показать пустое состояние с подсказкой |
| 3 | Частичные данные (нет фонда) | Показать доступные блоки, скрыть пустые графики |
| 4 | Неверный фильтр | Ответ 400 + сообщение "Неверный фильтр" |
| 5 | API недоступен | Показать error-state с кнопкой "Попробовать снова" |
| 6 | Ошибка синхронизации Robokassa/Prodamus/Tochka | Логировать ошибку, отправить алерт, не блокировать остальные источники и Mini App (работает с существующими данными) |
| 7 | Credentials источника невалидны | Вернуть ошибку и отключить синхронизацию этого источника до исправления |
| 8 | Утечка API-ключей в лог | Strict-check в logger: никогда не логировать переменные окружения, только метаинформацию |

## Итог
Mini App будет легким и безопасным, с готовыми аналитическими текстами и визуализациями, без AI-запросов из браузера. Это расширение позволит добавить платформу аналитики и многопользовательский доступ, сохранив текущую архитектуру FinAssist.
