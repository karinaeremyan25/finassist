# Feature: Интеграция Точки и Робокассы в Mini App (доходы, расходы, фонды, движения по фондам)

> **Проект:** FinAssist · Telegram Mini App (Vercel + Supabase)
> **Дата:** 2026-06-11
> **Приоритет:** High
> **Статус:** спецификация (к реализации)
> **Контекст:** приложение уже работает на Vercel, авторизация и дашборд готовы; в БД сейчас 12 операций (загружены вручную из выписки эквайринга). Нужно автоматическое наполнение из Точки и Робокассы + корректные фонды и движения по ним.

---

## 1. Цель

Сделать так, чтобы дашборд показывал **полную и автоматическую** картину:
- **Доходы** — все поступления (Робокасса, эквайринг, переводы на счёт Точки);
- **Расходы** — все списания со счёта Точки (по категориям);
- **Фонды** — реальные балансы фондов из Точки (налоговый, резервный, развития, благодарность, кредиты и т.д.);
- **Движения по фондам** — пополнения (из дохода) и **траты с фондов: куда и зачем** (на что потрачены деньги фонда).

Сейчас доход в приложении (341 828 ₽) — это только эквайринг. У бухгалтера за период — 857 015 ₽. Разница = поступления, которых нет в базе (Робокасса + переводы на счёт). Точка и Робокасса закрывают это.

---

## 2. Кто что отдаёт (модель источников)

| Источник | Тип | Что приносит | Механизм |
|----------|-----|--------------|----------|
| **Робокасса** | push (webhook) | разовые онлайн-оплаты курса/клуба (доходы ИП) | Robokassa POST'ит на ResultURL при каждой оплате |
| **Точка (расчётный счёт ООО/ИП)** | pull (OAuth API) | **всё движение по счёту**: входящие (доходы), исходящие (расходы), и **балансы фондов-копилок** | приложение опрашивает API Точки по расписанию |
| Эквайринг (Платформа ОФД) | файл-выписка | оплаты картой (доходы) | ручная/файловая загрузка (уже сделано разово) |

> Точка — главный источник: там и доходы, и расходы, и фонды. Робокасса — отдельный канал входящих платежей, который на счёт может приходить агрегировано, поэтому ловим её отдельно (детализация по плательщикам).

## 3. Реальная схема БД (важно — под неё всё мапим)

> Текущая боевая база отличается от первоначальных миграций. Реальные таблицы/колонки:

**`transactions`** — все операции (доходы и расходы):
`id, flow_type('income'|'expense'), amount(bigint, копейки), currency, amount_rub(bigint), fx_rate, entity_id, direction_id, category_id, source_id, occurred_at(timestamptz), description, external_id(text, дедуп), created_by(bigint), verified, needs_classification, deleted_at`

**`funds`** — фонды (баланс хранится ПРЯМО в колонке):
`id, name(текст названия), code, entity_id, balance(bigint, копейки), is_active, deleted_at`
Реальные коды фондов: `tax_ip, tax_ooo, reserve_ip, reserve_ooo, development_ip, development_ooo, personal`.
⚠️ Нет фондов «благодарность»/«кредиты»/«земля» — их нужно ДОБАВИТЬ (см. §7), т.к. они есть в твоих остатках из Точки.

**`fund_transactions`** — движения по фондам (пополнения и траты):
`id, fund_id, amount(bigint, + пополнение / − трата), description(зачем), created_by, created_at`

**`entities`**: коды `IP` (УСН 6%), `OOO` (УСН 15%), поле `tax_type`.
**`directions`**: коды `DPO` (Курс ДПО), `METANOIA` (Клуб).
**`sources`**: есть `prodamus, card_ip, rs_ooo, card_personal, card_foreign, cash`.
⚠️ **Нет источников `robokassa` и `tochka`** — добавить (см. §7).
**`categories`**: доход — `prodamus_course, prodamus_club, other_income`; расход — `marketing, video, contractors, rent, salary, taxes, bank_fees, other_expense`.

## 4. Как приходят ДОХОДЫ

### 4.1 Робокасса (webhook)
- Эндпоинт (Vercel serverless): `POST /api/webhooks/robokassa` — **уже реализован** (`src/services/integrations/robokassa.ts`), нужно проверить под Vercel.
- Robokassa в личном кабинете → Технические настройки → **Result URL** = `https://finassist-virid.vercel.app/api/webhooks/robokassa` (метод POST).
- Проверка подписи: `MD5(OutSum:InvId:Пароль#2[:Shp_*])` (реализовано, покрыто тестом).
- Маппинг → `transactions`:
  - `flow_type='income'`, `amount`/`amount_rub` = `toKopecks(OutSum)`, `currency='RUB'`,
  - `entity_id` = ИП (`IP`), `direction_id` = по продукту (DPO/METANOIA через `prodamus_product_mapping` или эвристику),
  - `source_id` = `robokassa`, `category_id` = `prodamus_course`/`prodamus_club`,
  - `external_id='robokassa_'+InvId` (дедуп), `occurred_at` = время уведомления.

### 4.2 Точка — входящие по счёту (pull)
- Из выписки Точки берём операции с `creditDebitIndicator='Credit'` (поступления).
- Маппинг → `transactions` (`flow_type='income'`), `source_id='tochka'`, `entity_id` по счёту (ИП или ООО), категория по описанию (платёж за курс → `prodamus_course`, прочее → `other_income`).
- Дедуп по `external_id='tochka_'+transactionId`.

> Антидубль с эквайрингом/Робокассой: если деньги с эквайринга приходят на счёт Точки агрегированной выплатой, помечать такие зачисления отдельной категорией (`acquiring_settlement`), чтобы не считать доход дважды. Решается правилом по описанию платежа.

---

## 5. Как приходят РАСХОДЫ

Расходы берём **из исходящих операций счёта Точки** (`creditDebitIndicator='Debit'`):
- Маппинг → `transactions` (`flow_type='expense'`), `source_id='tochka'`, `amount`/`amount_rub` = сумма списания.
- **Категоризация** (по контрагенту/назначению платежа Точки):
  - налоги/взносы → `taxes`; зарплата/исполнители → `salary`/`contractors`; реклама → `marketing`; аренда → `rent`; банковские комиссии → `bank_fees`; видео/продакшн → `video`; прочее → `other_expense`.
  - на старте — простая эвристика по ключевым словам в назначении; неуверенные → `needs_classification=true` (потом подтверждаешь в боте/приложении).
- Дедуп по `external_id='tochka_'+transactionId`.

➡️ Как только расходы появятся в `transactions`, **диаграмма автоматически покажет сегмент «Расходы»** (код уже готов, ему просто нужны данные), а KPI «Расходы» перестанет быть нулём.

## 6. ФОНДЫ и движения по ним (главная просьба: «траты с фондов — куда и зачем»)

### 6.1 Что такое фонд здесь
Фонд — «копилка» под цель: налоговый резерв, резервный, развитие, благодарность, кредиты и т.п. У фонда есть **баланс** (`funds.balance`) и **история движений** (`fund_transactions`).

### 6.2 Откуда берутся балансы фондов
Два способа (выбрать; рекомендуется A, раз «в Точке есть все фонды»):

- **A. Из Точки напрямую (рекомендуется).** В Точке фонды — это отдельные **счета-копилки/накопительные счета**. API Точки отдаёт список счетов и их балансы. Маппим каждый счёт-копилку → фонд (`funds.balance`) по соответствию `tochka_account_id ↔ fund`. Тогда балансы в приложении = ровно как в банке. Нужна таблица соответствия (см. §7) и поле `funds.tochka_account_id`.
  - Синхронизация: при каждом pull Точки обновляем `funds.balance` из баланса соответствующего счёта.

- **B. Считаем сами из распределения.** При каждом крупном доходе раскладываем % по фондам (налог/резерв/развитие/личное) → пишем `fund_transactions(+)` и увеличиваем `funds.balance`. Минус — расходится с фактом в банке. Использовать, только если в Точке фонды НЕ отдельными счетами.

### 6.3 Пополнение фонда (приход)
- Запись `fund_transactions`: `fund_id`, `amount = +сумма`, `description='Пополнение из дохода <дата>'`, `created_by`.
- Способ A: фиксируем по факту прихода на счёт-копилку в Точке. Способ B: по правилу распределения от дохода.

### 6.4 ТРАТА с фонда — «куда и зачем»
Когда деньги уходят ИЗ фонда (заплатили налог из налогового фонда, оплатили подрядчика из фонда развития, выплатили кредит из фонда «кредиты»):
- Запись `fund_transactions`: `amount = −сумма`, `description='<зачем>: <куда>'` (например «Налог УСН за 2 кв — в ФНС», «Оплата монтажёру — фонд развития»), `created_by`.
- Уменьшаем `funds.balance`.
- **Связь с расходом**: трата фонда обычно соответствует строке в `transactions` (расход). Чтобы видеть «куда ушли деньги фонда», связываем `fund_transactions` с расходной транзакцией. Для этого добавить поле `fund_transactions.source_transaction_id` (FK на `transactions.id`).
  - Тогда «куда» = контрагент/назначение из связанной расходной операции; «зачем» = `description` движения по фонду + категория расхода.

### 6.5 Как это определяется автоматически (из Точки)
- Если фонды — отдельные счета (способ A): исходящая операция со **счёта-копилки** = трата с фонда. Её назначение платежа = «куда» (контрагент), категория = «зачем». Маппим: `fund_transactions(−)` + расходная `transactions` со связью.
- Переводы **между** основным счётом и копилкой = внутреннее перемещение (пополнение/возврат фонда), НЕ доход и НЕ расход бизнеса (помечать `internal_transfer`, исключать из P&L).

### 6.6 Что показываем в приложении
- Экран **«Фонды»** (см. §8): по каждому фонду — название, текущий баланс, и история: пополнения (+) и траты (−) с подписью «куда/зачем».

## 7. Изменения в БД (миграции)

```sql
INSERT INTO sources (code, display_name, is_active) VALUES

  ('tochka',    'Точка (счёт)', true)

-- 2. Недостающие фонды (есть в остатках Точки, нет в БД)
INSERT INTO funds (name, code, balance, is_active) VALUES
  ('Фонд «Благодарность»', 'gratitude', 0, true),
  ('Фонд «Кредиты»',       'credit',    0, true),
  ('Фонд «Земля»',         'land',      0, true)

-- 3. Связь траты фонда с расходной операцией («куда ушли деньги»)
ALTER TABLE fund_transactions
  ADD COLUMN IF NOT EXISTS source_transaction_id UUID REFERENCES transactions(id) ON DELETE SET NULL;

-- 4. Соответствие фонд ↔ счёт-копилка Точки (для способа A)
ALTER TABLE funds
  ADD COLUMN IF NOT EXISTS tochka_account_id TEXT;

-- 5. Тип категории для внутренних переводов и расчётов эквайринга (чтобы не задваивать доход)
INSERT INTO categories (code, display_name, flow_type, accounting_type, is_active) VALUES
  ('internal_transfer',     'Внутренний перевод (фонд)', 'expense', 'transfer', true),
  ('acquiring_settlement',  'Зачисление эквайринга',     'income',  'transfer', true)
ON CONFLICT (code) DO NOTHING;
```

Применяется через существующий раннер: `npm run migrate` (файл `db/migrations/006_*.sql`).

## 9. Конфигурация / секреты (Vercel env)

```
# Робокасса
ROBOKASSA_MERCHANT_LOGIN=...
ROBOKASSA_PASSWORD=...           # Пароль #2 (для подписи webhook)

# Точка (OAuth 2.0)
TOCHKA_CLIENT_ID=...
TOCHKA_CLIENT_SECRET=...
TOCHKA_REDIRECT_URI=https://finassist-virid.vercel.app/api/tochka/callback
```

> Точка использует **authorization_code + consent** (не client_credentials). Нужно реализовать callback-эндпоинт `GET /api/tochka/callback`, который примет `code`, обменяет на токены и сохранит `refresh_token` в `settings`/отдельную таблицу. Redirect URL в кабинете Точки = адрес выше (его как раз спрашивала форма регистрации приложения).

## 10. Синхронизация (расписание)

- **Робокасса** — реального времени (webhook на каждую оплату).
- **Точка** — pull по расписанию. На Vercel Hobby cron ограничен (1 раз/сутки), поэтому варианты:
  - cron-задача в боте на VPS (если бот развёрнут) — каждые 30 мин;
  - либо Vercel Cron (1×/день) + кнопка «Обновить из Точки» в приложении (ручной вызов синка);
  - дедупликация по `external_id` делает повторные синки безопасными.

## 11. Edge cases

| # | Ситуация | Поведение |
|---|----------|-----------|
| 1 | Эквайринг и Робокасса оба пишут один платёж | дедуп по `external_id`; зачисление эквайринга на счёт Точки → категория `acquiring_settlement`, не считается доходом повторно |
| 2 | Перевод между счётом и копилкой Точки | `internal_transfer`, исключается из P&L (не доход/не расход) |
| 3 | Расход без явной категории | `needs_classification=true`, подтверждается вручную |
| 4 | Невалидная подпись Робокассы | 400, не пишем транзакцию (реализовано) |
| 5 | Истёк токен Точки | авто-refresh; при провале — алерт, синк пропускается |
| 6 | Трата с фонда без расходной операции (нал/прочее) | `fund_transactions(−)` без `source_transaction_id`, «куда» из `description` |

## 12. План реализации (по шагам)

1. **Миграция 006** (§7): источники, фонды, `fund_transactions.source_transaction_id`, `funds.tochka_account_id`, категории-переводы. Применить `npm run migrate`.
2. **Робокасса**: проверить webhook на Vercel, прописать Result URL в кабинете, тестовая оплата → доход в базе.
3. **Точка OAuth**: реализовать `/api/tochka/callback`, хранение токенов; завершить регистрацию приложения в Точке с Redirect URL.
4. **Точка sync**: pull выписки → доходы/расходы (§4.2, §5) + балансы фондов и движения (§6) → `transactions`/`funds`/`fund_transactions`.
5. **Mini App**: «Счета»→«Фонды», экран фондов с движениями, `GET /api/analytics/funds`, расходы в donut.
6. **Проверка**: доход сойдётся с бухгалтером (≈857 015 ₽), появятся расходы, балансы фондов = как в Точке.

## Итог
Точка даёт полную картину (доходы + расходы + фонды + движения), Робокасса — детализацию онлайн-оплат. Деньги фонда отслеживаются через `fund_transactions` со связью на расходную операцию — поэтому по каждому фонду видно **сколько, куда и зачем** ушло. Всё мапится на реальную схему БД; диаграмма и экран «Фонды» наполняются автоматически после подключения источников.
